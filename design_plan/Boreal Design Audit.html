Design audit · v1

Boreal is quietly excellent — but it's getting in its own way.

You've built a finance app with real visual taste: oklch tokens, a coherent nordic-calm palette, tabular numerics, an aurora gradient that earns its keep. But the app is also overgrown. Multiple "v2" components live next to v1 leftovers, the dashboard tries to do nine things at once, and the marketing/in-app aesthetics speak two different languages. This audit walks every surface and tells you what to keep, what to trim, and what to redesign.

01 — OverviewExecutive summary

If you do nothing else from this audit, do these seven things. They're cheap, they compound, and they fix the loudest problems first.

  1. 1 Pick one identity. The landing page is dark/nocturnal and the app is light/paper. They share a logo and not much else. Either ship a light landing that previews the actual product, or ship a dark app theme that matches the marketing — but stop selling people one product and delivering another.
  2. 2 Cut the dashboard in half. Today it's: headline → insights → aurora hero → 3 flow cards → net worth chart → spend doughnut → budgets → accounts → recurring → goals → 12-month bars → forecast chart. Twelve modules competing for a glance. Pick a primary KPI (net worth or this-month savings, not both), demote everything else into a second-tier grid, and move recurring/forecast/goals to their own sub-tab.
  3. 3 Kill the aurora gradient in the app (keep it on the logo). Right now the dashboard hero, the empty-states, and the brand mark all use the same blue→teal→green gradient. It's loud, it doesn't match the rest of the calm light theme, and it makes net worth feel like a marketing banner. Replace with a flat surface and a single accent line; reserve the aurora for the 28-pixel logo mark and onboarding only.
  4. 4 Consolidate the "v2" components. Your CSS has .kpi and .dash-flow .f-card; .recur-row and .recur-v2; .budget-v2 with no v1 to compare; .insights and .insights-v2. Ship one canonical component for each pattern and delete the rest. This is the single biggest source of design entropy in the codebase.
  5. 5 Re-do the sidebar for density. Eight nav items + a theme toggle + a user row at 13.5px is borderline cluttered. Group "Workspace" and "Automate" works; merge Year review into Dashboard as a tab, and move Rules into Import as a secondary tab. Targets: 5 top-level items, one settings slot.
  6. 6 Fix copywriting voice. The landing is confident ("The free personal finance app built for Canadians"), the app is hesitant ("No goals yet", "Not enough data yet"). Empty states should sell the next action, not describe absence. Examples below.
  7. 7 Build a proper mobile shell. The 1100px breakpoint collapses the sidebar to a 64px icon rail, but the topbar still has a 240px search input, a 110px month chip, and a notification bell — they overflow. Below 720px there's no mobile nav at all. Implement a bottom tab bar at <720px and turn the topbar into a sticky page header.

Color tokens — fix

PolishDS-01

The dark-mode accent is too saturated

Light mode accent is oklch(38% .075 165) — a confident, muted pine. Dark mode jumps to oklch(72% .13 165) — almost a neon mint, especially against the deep navy bg. It makes the dark theme feel "vibrant SaaS" while light feels "considered tool".

Fix. Dial dark accent to oklch(68% .09 165). Keep the chroma close to light mode so the two themes feel like the same product.
PolishDS-02

--neg is named negatively but used for expenses (which aren't bad)

You have --neg: oklch(48% .005 240) — a near-black neutral — that you apply to expense amounts. Semantically that's fine (expenses shouldn't blare red), but the variable name says "negative" which suggests "wrong". And then expense amounts in the table use --ink-1 directly, not --neg, so the token is half-orphaned.

Fix. Rename to --money-out; rename --pos to --money-in. Use them consistently everywhere amounts appear (table, drawer, dashboard).

Type — keep, but tune

Inter Tight + JetBrains Mono is a strong pair: Inter Tight's negative letterspacing makes large numbers feel typographic instead of spreadsheet-ish, and the mono is reserved appropriately for keyboard shortcuts and dates. What's missing is a display face for moments where you want voice — the dashboard headline ("You saved $1,240 · 26% savings rate"), empty-state titles, the year-review hero. Today those use the same Inter Tight 500 as nav, which flattens the hierarchy.

Proposed type scale

display / 48 / 500 You saved $1,240 h1-page / 26 / 600 Transactions h2-card / 13 / 500 / case Where it went body / 13.5 / 450 Three groceries trips at Loblaws this week. num-kpi / 28 / 500 / tab $12,480.20 label / 11 / 600 / upper Income

Add Fraunces (or any modern serif like Newsreader/Source Serif) as your display face — used only for: dashboard headline, year-review title, onboarding, empty-state titles, and the landing H1. This gives Boreal a voice that distinguishes it from the dozens of YNAB/Copilot/Monarch-style apps.

Spacing & radii — keep

Your ladder (6/10/14/20 for radii, padding generally 16/20/24) is consistent and pleasant. No notes.

Specific issues

CriticalL-01

Marketing and product use different visual languages

The landing is forest-dark, mockup-as-window-chrome, marketing-SaaS in feel. The actual app is paper-light, sans-chrome, productivity-tool in feel. A visitor signs up expecting the dark hero experience and lands in something that looks unrelated.

Fix. Re-shoot the landing in the light theme. Make the hero "preview" actually look like the app: same sidebar, same KPI cards, same fonts. Sell what you ship.
HighL-02

Feature grid is generic

"Auto-categorize everything / Budgets & goals / Net worth / Privacy-first" — these are table-stakes claims with abstract icons. Anyone reading them has seen them on five other apps this week.

Fix. Replace abstract icons with cropped screenshots of the actual feature. "Auto-categorize" → show a row going from UNCATEGORIZED → Groceries. "Net worth" → show the line chart. This converts.
HighL-03

Missing differentiation section

Your README has the best sales pitch in the repo: "Zero cost. Fully local. No subscriptions, no cloud, no ads, no tracking." That's not on the landing page.

Fix. Add a comparison section between Features and Banks: Boreal vs Monarch ($14.99/mo) vs Copilot ($13/mo) vs Mint (sunset). Three columns. Price · cloud-required · CSV import · Canadian banks. You win every row.
MedL-04

The "browser chrome" mockup is dated

Three traffic-light dots is the most-overused mockup frame on the internet (every YC company since 2018). It's also misleading — Boreal is a web app, not a desktop app.

Fix. Either ditch the chrome entirely and let the screenshot bleed, or use a tilted "screen at an angle" treatment to make it feel like a product not a website screenshot.

Topbar — be context-aware

Right now the topbar shows the month chip on every view even though only Dashboard, Transactions, Budgets, and Year actually filter by month. On Settings or Rules the chip is dead UI. The CSS already has .topbar.no-month — use it.

  • Lose the notifications bell unless you have a real notifications system. A bell with no badge is anxiety-inducing chrome.
  • Add a contextual action button on the right — "Add transaction" on the Dashboard/Transactions views, "Add account" on Accounts, "Import file" on Import. Currently this lives inside each page header which means it shifts position as you scroll.

07 — ChromeFAB, undo toast, bulk dock

You have three competing bottom-of-screen surfaces:

  • FAB (mobile only, 48px, bottom-right, ink-1 colored "+") — fine, conventional.
  • Undo toast (centered, pill-shaped, ink-1 background, "Undo" button + dismiss) — well-designed.
  • Bulk dock (centered, pill-shaped, ink-1 background, "N selected" + actions) — also well-designed, but visually identical to the undo toast.

If a user deletes 5 transactions while having them selected, both show up — same color, same position, same shape, and the CSS even has a special :has() rule to push the toast up when the dock is open. That's a smell.

Recommendation

Give them distinct identities: bulk dock stays centered & ink-1 (it's a tool), undo toast moves to bottom-right corner with a different shape (rounded rectangle, not pill) and a lighter elevation. That way they can co-exist visually without clashing, and a user instantly knows which is which.

From 9 modules to 4 zones

Zone 1 — The headline (full width).
Keep the comparative headline ("You saved $1,240 · 26% savings rate"). Set it in Fraunces, 32px. This is the answer the user wakes up to.

Zone 2 — Hero KPI + trend (full width, flat surface, no aurora).
Pick one primary metric: Net Worth. Big number, 14-day spark, "vs last month" delta. Make this feel like the front page of a financial newspaper, not a SaaS dashboard.

Zone 3 — The 4-card grid (2x2).
Cells: Spending breakdown Budget pacing (top 3) Accounts Recurring radar
One canonical card component, same height, same padding, same H3 style. Each has a "See all →" link in the corner.

Zone 4 — The trend strip (full width).
A single income vs expenses bar chart, 12 months, with the optional forecast extension. Goals and insights move out of dashboard entirely — see below.

Where everything goes

18 — Cross-cuttingModals & drawers

You have two overlay patterns: modals (centered card, backdrop blur) and the transaction-edit drawer (right-side slide-in). Plus a separate "drawer-edit" with bespoke styling for the edit experience.

Keep
  • Right-side drawer for transaction edit is the right pattern (don't lose context)
  • Drawer header with editable name + dot separator + amount type pill
  • Category picker inside drawer with search + grouped options — very nice
  • "Apply to similar" learn-callout — magical when it triggers
  • Expandable "Advanced" section keeps the drawer scannable
Change
  • Modal vs drawer rules aren't consistent — "Add transaction" is a modal, "Edit transaction" is a drawer
  • Modal width 480px, drawer width 480/520px — interchangeable in practice
  • Modal backdrop is oklch(15% .02 240 / 0.4); drawer backdrop is 0.3 — minor inconsistency
Fix
  • No focus trap visible in the code — Tab inside a drawer can escape to the page
  • Drawer animation is transform-only — escape-key handling needs verification
  • Stacking: opening a category picker inside a drawer inside a modal — does z-index hold?

Pick one mental model. Drawer = editing an existing entity (transaction, schedule, rule, account). Modal = creating a new one or a confirmation. This is a 30-minute cleanup that pays off forever.
